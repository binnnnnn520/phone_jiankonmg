# Deployment Checklist

This checklist is for the product operator, not ordinary users.

Use it before calling a remote deployment ready for real two-phone validation.

## 1. Runtime configuration

- `SIGNALING_HOST` is set for the local bind address
- `SIGNALING_PORT` is set
- `PUBLIC_SIGNALING_HTTP_URL` points to the public HTTPS signaling endpoint
- `ICE_SERVERS_JSON` contains the intended STUN/TURN entries
- `ROOM_TTL_SECONDS` is set
- `PIN_MAX_ATTEMPTS` is set
- `VITE_SIGNALING_HTTP_URL` points to the same public HTTPS signaling endpoint
- `VITE_SIGNALING_WS_URL` points to the public WSS endpoint
- `VITE_PUBLIC_VIEWER_URL` points to the public viewer app URL
- `VITE_PREFERRED_CONNECTION_MODE` is `auto` for production-like builds

## 2. Build outputs

- `packages/shared/dist` exists
- `packages/server/dist` exists
- `packages/web/dist` exists
- Root `npm run build` completes without errors

## 3. Public app and signaling reachability

- Public app URL loads over HTTPS
- Public signaling health check returns `{"ok":true}`
- Public WebSocket endpoint is reachable at `/ws`
- QR codes generated by the camera page use the intended public viewer URL, not `localhost`

## 4. Media path prerequisites

- The deployment exposes HTTPS and WSS, not plain HTTP/WS for remote use
- `ICE_SERVERS_JSON` is valid JSON
- At least one STUN server is configured
- TURN is configured for real remote-network validation
- TURN credentials are not missing or placeholder values
- Any copied STUN-only sample config has been replaced with the real intended STUN/TURN set

## 5. Product behavior checks

- Camera page starts and requests camera permission
- Camera page requests microphone permission
- If microphone is granted, the camera UI shows environment audio is live
- If microphone is denied, the camera UI falls back to video-only
- Viewer page can scan or open the room link, enter a PIN, and connect
- Viewer page can unmute audio and hear environment sound when available
- Stopping the camera session ends the viewer session cleanly

## 6. Real-device validation

- Same-Wi-Fi validation completed
- Remote validation completed with the viewer on mobile data or another external network
- At least one session confirms whether relay was used
- Wrong PIN behavior was checked
- Expired room behavior was checked
- A longer foreground session was observed for reconnect behavior

See [manual-remote-validation.md](../testing/manual-remote-validation.md) for the detailed device checklist.

## 7. Logging and privacy checks

- Production logs do not store SDP payload contents
- The signaling service does not store video, screenshots, or audio payloads
- Room state remains short-lived
- Camera-side monitoring status remains visible in the UI

## 8. Release gate

Do not call the deployment production-ready until:

- HTTPS app hosting works
- WSS signaling works
- TURN is configured and validated on a real remote network
- The two-phone flow succeeds end to end on real devices
